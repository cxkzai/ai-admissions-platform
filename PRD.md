## 0. 项目一句话

**AI-Admissions Platform 是一个面向中小教培机构（**V1 聚焦 K12 少儿编程教育**）的 AI 招生顾问平台**，通过多角色智能体 + RAG 知识库 + 自动化工作流，覆盖"线索获取 → 需求挖掘 → 试听转化 → 老带新裂变"招生全链路，让一个招生主管 + 一线顾问的团队效能翻倍。

## 1. 项目概述

### 1.1 业务背景

**V1 聚焦 K12 少儿编程教育**（8-15 岁），可平滑扩展至数学思维、艺术素养等其他赛道。教培行业的招生高度依赖"人海战术 + 话术 + 速度"，少儿编程赛道尤其如此：

| 行业痛点 | 现状 | 代价 |
|---|---|---|
| **深夜/周末线索流失** | 家长 22:00 留言，顾问第二天才回 | 30%+ 线索流失 |
| **新人顾问成单率低** | 没有体系化话术，靠"师傅带" | 3 个月才能上手 |
| **知识资产散落** | 课程介绍、师资、价格散落在 5+ 地方 | 顾问口头错误频发 |
| **跟进节奏失控** | 试听后忘了跟进、跟进话术千人千面 | 试听转化率 20%~35% |
| **老板看不见数据** | Excel + 微信群汇报，数据滞后 1 周 | 决策慢、调整难 |

**而 AI 招生顾问恰好可以解决其中 80% 的问题**：7×24h 接待新人、按标准化流程转化、把话术/课程/师资沉淀成可检索的知识库。

### 1.2 项目目标

为求职者本人（投递"AI 应用工程师"岗位）打造一个**可公网访问、可演示、可交付的端到端项目**，证明以下能力：

1. ✅ **多角色智能体设计**：能用 Prompt + 工具调用搭出真实可用的 Agent
2. ✅ **RAG 知识库工程**：能完成文档清洗 → 切片 → 向量化 → 检索 → 评估闭环
3. ✅ **工作流自动化**：能编排多步骤业务流（线索 → 跟进 → 转化）
4. ✅ **企业 IM 对接**：能打通飞书/企微 Webhook，做消息推送
5. ✅ **全栈交付能力**：Python 后端 + Vue 3 前端 + 完整部署文档
6. ✅ **项目交付意识**：有 README、Demo 截图、操作手册、架构图

### 1.3 核心价值主张

| 对谁 | 解决什么 | 量化收益 |
|---|---|---|
| **家长** | 深夜咨询也能秒回，回复专业有温度 | 首响时长 < 30s |
| **课程顾问** | 不用记课程/价格，话术由 AI 推荐 | 跟单效率 +50% |
| **招生主管** | 招生漏斗实时可见，AI 异常告警 | 数据滞后从 1 周 → 实时 |
| **机构老板** | 沉淀知识资产，新人培养周期缩短 | 上手周期 3 月 → 1 月 |

### 1.4 名词解释

| 术语 | 含义 |
|---|---|
| **Agent（智能体）** | 基于 LLM、能调用工具、完成特定任务的 AI 角色。本项目有 4 类 Agent。 |
| **RAG（检索增强生成）** | LLM 在回答前先检索知识库，拿到相关文档再回答，减少幻觉。 |
| **知识库** | 把课程介绍、师资、FAQ、话术文档结构化、向量化后存储，可被 Agent 检索。 |
| **工作流** | 多个 Agent + 工具 + 条件分支组成的业务流程（如"新线索 → 24h 内 3 次触达"）。 |
| **Function Calling** | LLM 调用外部工具（如查课表、发飞书消息、写数据库）的能力。 |
| **Web 演示台** | 公网部署的 Demo，供面试官点击体验，是本项目的"产品门面"。 |

### 3.2 模块 1：多角色智能体（核心）

#### 3.2.1 招生顾问 Agent（对外主接待）

- **角色定位**：机构对外"首席咨询师"，承担 60% 接待任务
- **核心能力**：
  - 多轮对话挖掘需求（年龄、目标、预算、时间）
  - 情绪识别：识别家长焦虑/纠结/对比心理，调整话术
  - 智能推荐课程（基于 RAG 检索课程库）
  - 自动预约试听（调用课程表工具 + 推送预约链接）
  - 留资引导（手机号/微信）
- **Prompt 设计要点**：
  - Persona：资深教育顾问，10 年经验
  - Tone：温暖、专业、有边界感（不夸大、不贬低同行）
  - Guardrails：禁用承诺、禁用价格"最低"、禁用医疗/心理类诊断
- **工具调用**：
  - `search_courses`：检索课程库
  - `search_teachers`：检索师资库
  - `check_schedule`：查询试听课表
  - `book_audition`：预约试听
  - `send_to_feishu`：推送飞书消息给顾问

#### 3.2.2 课程顾问 Agent（专业咨询）

- **角色定位**：辅助一线顾问，提供深度课程咨询
- **核心能力**：
  - 课程对比分析（A 课 vs B 课）
  - 师资介绍生成
  - 竞品差异化话术生成
  - 报价单生成（PDF）
- **使用方式**：Web 端对话界面，一线顾问手动发起

#### 3.2.3 教务老师 Agent（试听跟进）

- **角色定位**：负责试听课后跟进，提升转化
- **核心能力**：
  - 试听反馈分析（基于家长/孩子的评价文本）
  - 跟进话术生成（按跟进阶段：第 1 天/第 3 天/第 7 天）
  - 流失风险识别（连续 3 次未回复/负面反馈）
- **触发方式**：试听课结束 + 定时任务

#### 3.2.4 内部知识助手 Agent（对内）

- **角色定位**：员工"百事通"，沉淀 SOP 与制度
- **核心能力**：
  - SOP 检索（如"试听课前要做什么"、"退费流程"）
  - 制度问答（如"年假怎么请"、"提成怎么算"）
  - 跨库检索（课程/师资/SOP/案例统一检索）

### 3.3 模块 2：知识库与 RAG

#### 3.3.1 知识库分类

| 库名 | 数据来源（示例） | 切片策略 | 检索优先级 |
|---|---|---|---|
| **课程库** | 课程介绍 PDF、官网课程页 | 按课程名切片（200-500 token） | 最高 |
| **师资库** | 老师简历、教学成果 | 按老师切片 | 高 |
| **案例库** | 学员故事、提分案例 | 按案例切片（500-1000 token） | 中 |
| **FAQ 库** | 家长高频问题 | 整条 FAQ 作为切片 | 最高 |
| **话术库** | 优秀顾问销售录音转写 | 按场景切片（开场/挖需/关单） | 中 |
| **SOP 库** | 内部制度、流程文档 | 按章节切片 | 中（内部使用） |

#### 3.3.2 RAG 核心流程

```
文档输入 → 解析（PDF/DOCX/HTML/MD）
        → 清洗（去广告/格式化/去重）
        → 切片（按语义/按长度/按结构）
        → 向量化（Embedding API）
        → 入库（Chroma / pgvector）
                          ↓
用户提问 → Query 改写（结合上下文）→ 向量化 → 相似度检索 Top-K → 重排序（Rerank）
                          ↓
Prompt 拼接：系统提示词 + 检索结果 + 用户问题 → LLM 生成回答
                          ↓
回答输出 + 引用来源标注 + 置信度提示
```

#### 3.3.3 RAG 质量评估

- 提供 **评测数据集**（50+ 标注问答对）
- 指标：命中率（Recall@5）、答案相关性、人工评分
- 提供 **评测脚本**，可在 CI 中运行

### 3.4 模块 3：提示词模板库

#### 3.4.1 功能

- **模板分类**：按场景（招生/续费/退费/活动）+ 按角色（顾问/教务/主管）
- **模板编辑**：Web 端可视化编辑器（Markdown）

- **A/B 测试**：同一场景 2 个版本，记录转化效果
- **导入导出**：JSON 格式，方便沉淀到 SOP 库

#### 3.4.2 内置模板示例

- 续费提醒
- 老带新邀约
- 流失挽回

### 3.5 模块 4：工作流引擎

#### 3.5.1 招生全链路工作流（核心）

```
新线索进入
   ↓
【招生顾问 Agent】首响问候 + 基础挖需
   ↓
[分支判断] 是否留资？
   ├─ 是 → 进入培育池
   └─ 否 → 6h 后 AI 主动再触达（最多 3 次）
   ↓
【课程顾问 Agent】深度需求挖掘 + 课程推荐
   ↓
[分支判断] 是否预约试听？
   ├─ 是 → 进入试听池
   └─ 否 → 24h 后进入流失预警池
   ↓
试听课当天 → 教务老师人工接待 + AI 辅助话术
   ↓
试听后 24h → 【教务老师 Agent】反馈收集 + 初步跟进
   ↓
试听后 3 天 → 第 2 次跟进
   ↓
试听后 7 天 → 第 3 次跟进（最后窗口）
   ↓
[分支判断] 是否成交？
   ├─ 是 → 进入学员池，触发老带新培育
   └─ 否 → 进入长尾池，每月 AI 触达 1 次
```

#### 3.5.2 可视化编排（V1 可用简化版）

- **V1**：YAML/JSON 定义工作流节点，前端展示流程图（只读）
- **V2**：拖拽式编排（dnd-kit），用户自定义工作流
- **触发方式**：手动 / 定时 / Webhook / 事件

#### 3.5.3 飞书/企微推送节点

- 工作流中可配置"飞书群机器人"节点，自动推送消息
- 支持 Markdown 卡片格式

### 3.8 模块 7：与开源智能体平台的关系（Dify / Coze / 扣子 / FastGPT）

> ⚠️ **为什么必须有这一节**：岗位 JD 加分项明确提到"有 Coze/Dify/扣子/FastGPT 智能体或工作流实际案例"。自研项目若不主动说明与这些平台的关系，容易被面试官误判为"不了解行业生态"。本节明确自研定位 + 提供兼容方案，证明对主流平台也有深度理解。

#### 3.8.1 为什么自研而非基于 Dify/Coze 二次开发？

| 维度 | 自研（本项目） | 基于 Dify/Coze 二次开发 |
|---|---|---|
| **代码原创度** | 100% 自己写，展示全栈能力 | 主要靠平台框架，原创度低 |
| **Prompt 工程深度** | 完全可控，可做精细优化 | 受平台 DSL 约束 |
| **Function Calling** | 完全自定义协议 | 平台固定工具集 |
| **RAG 灵活性** | 自定义切片/重排序/评测 | 平台默认实现，黑盒 |
| **部署灵活性** | 任意云、私有化、轻量 | 通常需要平台服务 |
| **面试官印象** | "能造轮子" | "会用工具" |
| **上手速度** | 慢（4-6 周） | 快（1-2 周） |

**结论**：选择自研 = 用时间换深度，换取更强的"AI 应用工程师"能力证明。

#### 3.8.2 对开源智能体平台的兼容设计（差异化亮点）

虽然不基于 Dify/Coze 二次开发，但本项目主动提供 **3 项兼容能力**，证明对生态的熟悉度：

1. **工作流导出兼容**
   - 把本项目 YAML 定义的工作流，导出为 Dify 的 YAML DSL 格式
   - 客户如果后期想切到 Dify，能平滑迁移
   - 实现位置：`apps/api/app/services/workflow/exporters/dify_exporter.py`

2. **提示词模板兼容**
   - 支持导入 Coze 的 Prompt 模板（JSON 格式）
   - 支持导出为 OpenAI/Anthropic 双格式
   - 实现位置：`apps/api/app/services/prompt/importers/coze_importer.py`

3. **知识库数据互通**
   - 支持从 Dify 导出的知识库 JSON 一键导入
   - 支持导出到 LangChain / LlamaIndex 通用格式
   - 实现位置：`apps/api/app/services/rag/importers/`

#### 3.8.4 在面试中如何讲这个话题

**面试官可能问**："你了解 Dify/Coze 吗？为什么不用它们？"

**推荐回答**：
> "我深入使用过 Dify 和 Coze（**这里要演示一个截图或本地跑过的截图**）。它们的优势是开箱即用、工作流可视化编辑器完善。但对于**有定制 RAG 和复杂业务逻辑**的场景，平台会有约束。所以我选择自研，关键能力（如 RAG 切片策略、Function Calling 协议、工作流引擎）都自己实现。

### 5.4 关键数据流

```

   ↓ 用户发送消息
[FastAPI /api/v1/chat/message]
   ↓
[Agent Service]
   ├─ 加载会话上下文（Redis）
   ├─ 调用 LLM（Claude API）进行意图识别
   ├─ 如需检索 → 调用 RAG Service

   │     ├─ 向量化 → Chroma 检索 Top-K

   │     ├─ search_courses / search_teachers / book_audition 等
   │     └─ 工具结果回填 LLM
   ├─ 生成最终回复
   └─ 保存会话日志 + 推送飞书通知
   ↓
[返回用户 + 飞书群]
```

   ├─ 调用 Embedding API
   ├─ 写入 Chroma（含元数据：库名/来源/时间）

## 10. 风险评估与应对

| 风险 | 概率 | 影响 | 应对策略 |
|---|---|---|---|
| **LLM API 成本超预算** | 中 | 中 | 用 DeepSeek 作主力（便宜），Claude 作评测基准；缓存常见问答 |
| **RAG 命中率不达标** | 中 | 高 | 用 bge-m3 等强 Embedding；增加 Query 改写；准备人工标注评测集 |
| **Demo 部署不稳定** | 低 | 高 | 部署双备份（Railway + Render）；本地 Docker 可一键启动 |
| **演示时 LLM 限流** | 低 | 中 | 多 Key 轮询；预热常见问答缓存；提供 Mock 模式 |
| **微信/飞书接入复杂** | 中 | 中 | V1 用 Mock 演示，V2 接真实 Webhook；提供详细文档 |
| **简历筛选被刷** | 中 | 高 | 项目名 + 描述要打中"AI 应用工程师"关键词；录讲解视频 |
